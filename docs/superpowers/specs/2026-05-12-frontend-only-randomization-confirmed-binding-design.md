# 前端临时随机与确认入组绑定设计

日期：2026-05-12  
范围：MotionCare 管理端项目分组看板、研究项目患者关系接口、患者选择与确认入组规则。  
状态：待用户 review 后进入实现计划。

## 背景

现有实现和早前文档仍残留 `pending` 入组、草案、患者池、批次或患者与项目预绑定等语义。产品规则需要进一步收敛：

- 患者与项目之间不存在独立的“加入/绑定”关系。
- 只有患者被确认加入某项目的某个分组后，才形成正式绑定。
- “患者池”不是产品实体，只是表达全量患者选择范围的界面概念。
- 批次、草案、未确认绑定都不应存在。
- 未确认随机结果只发生在前端页面内存中，不落库；刷新、切换项目或关闭浏览器后直接丢弃。

## 目标语义

`ProjectPatient` 只表达一件事：某患者已经确认加入某研究项目的某个分组。创建 `ProjectPatient` 即代表已确认入组，不再使用它表达待确认、草案或预加入状态。

项目分组看板中存在两类患者展示：

- 已确认患者：来自后端 `ProjectPatient`，在所属组内置灰展示，不参与随机，不可拖拽改组，只能通过“从本项目移除”解绑。
- 本次随机患者：来自当前页面勾选后前端计算出的临时结果，未确认前只存在于前端内存，可移除或被重新随机覆盖。

## 数据模型与接口

后端保留 `StudyProject`、`StudyGroup`、`Patient`、`ProjectPatient`。`ProjectPatient.grouping_status=pending` 不再作为业务状态使用；实现阶段可选择删除字段，或短期内保留字段但所有创建路径都写入 `confirmed`，并在序列化和接口语义上不暴露 pending 概念。

`GET /api/studies/project-patients/?project={id}` 只返回已确认入组患者。它不返回临时随机结果，因为这些结果不在服务端存在。

确认入组接口使用项目维度提交前端临时结果：

```json
{
  "assignments": [
    { "patient_id": 1, "group_id": 10 },
    { "patient_id": 2, "group_id": 11 }
  ]
}
```

推荐沿用并改造 `POST /api/studies/projects/{id}/confirm-grouping/`。该接口不再确认已有 pending 行，而是批量创建已确认 `ProjectPatient`。服务端必须校验：

- 所有患者存在。
- 所有分组存在、启用，且属于当前项目。
- 患者尚未在当前项目中确认入组。
- 同一次请求中同一患者不能重复提交。

确认成功后返回创建结果和确认数量。若任一校验失败，接口应返回明确错误，不创建部分数据；实现阶段用事务保证原子性。

原 `randomize`、`reset-pending` 等会创建或修改 pending 关系的后端语义应移除或停用。随机算法迁移到前端纯函数中，输入为已勾选且未确认患者、启用分组及其比例，输出为页面内临时分配。

已确认患者从组内删除继续使用现有 `POST /api/studies/project-patients/{id}/unbind/`。删除成功后，该患者不再是当前项目已确认患者，会重新变为可勾选对象。

## 前端看板交互

患者选择区域建议改名为“全量患者”或“患者选择”，避免把“患者池”理解成实际产品关系。该区域展示全量患者：

- 当前项目已确认患者置灰禁选，并标注其已确认分组。
- 未在当前项目确认的患者可勾选。
- 勾选状态只表示“本次随机输入”，不是入组或绑定。

点击“随机分组”时，前端只使用当前勾选且未确认的患者生成临时分配。未勾选患者不参与随机。已确认患者不参与随机。再次点击随机时，用当前勾选集合重新生成临时分配，并覆盖上一轮未确认结果。

组列永远展示该组已确认患者，同时追加展示本次随机患者。已确认患者置灰、不可拖拽、不隐藏；本次随机患者以正常或待确认样式展示，可从本次结果中移除。

点击“确认分组”时，前端把当前临时分配提交给确认接口。成功后：

- 清空页面内临时分配。
- 清空对应勾选状态。
- 刷新已确认患者列表。
- 新确认患者进入组列已确认展示，并变为禁选。

刷新页面、切换项目、离开页面或关闭浏览器时，未确认临时分配直接丢弃，不做恢复提示，不写本地存储。

## 错误处理

随机分组在前端完成。若没有可随机患者、没有启用分组、分组比例无效，前端直接提示，不请求后端。

确认接口应作为最终一致性校验边界。页面停留期间如果患者已经被他人确认、分组被删除或停用、患者记录不存在，后端拒绝请求并返回明确文案。前端收到错误后刷新患者与分组数据，让置灰禁选状态回到最新。

“从本项目移除”属于高风险操作，继续使用二次确认。确认文案需说明会解除该患者在本项目下的已确认入组关系，并按既有服务端策略处理关联 CRF、访视、处方或导出数据。

## 测试范围

后端测试：

- 确认接口按 `assignments` 批量创建已确认 `ProjectPatient`。
- 确认接口拒绝已在当前项目确认的患者。
- 确认接口拒绝跨项目分组、未知患者、未知分组、停用分组和重复患者。
- 项目患者列表只返回已确认绑定，不表达 pending 或草案。
- 解绑后患者不再出现在当前项目已确认患者列表。

前端测试：

- 全量患者区域展示所有患者，当前项目已确认患者置灰禁选。
- 随机只对当前勾选且未确认患者生成页面内临时分配。
- 随机不会调用后端落库接口。
- 再次随机会覆盖上一轮临时分配。
- 组列同时展示已确认患者与本次随机患者。
- 已确认患者在组列置灰展示且不隐藏。
- 确认时提交 `assignments`，成功后清空临时分配并刷新已确认列表。

## 非目标

- 不做批次历史、草案保存、本地暂存或跨浏览器恢复。
- 不把患者池建模为后端实体。
- 不支持未确认随机结果的多人协作编辑。
- 不在本次设计中改变 CRF、访视、处方的解绑治理策略，只沿用既有解绑服务。
