## 背景

项目详情页已经回归为单一分组看板，并且随机结果与分组占比都已收敛为前端页面草案：只有点击确认后才保存。本次需要进一步优化项目详情页的动作区域，并补齐项目生命周期中的“完结”能力。

当前问题：

- `随机分组`、`确认分组` 在看板内部，`新增分组` 在项目详情页标题右上，动作分散。
- 看板仍有“占比与本轮随机结果仅在确认后保存。”说明文案，页面变得冗余。
- 项目已有 `archived` 状态，但产品侧需要表达为“已完结”，并在完结后阻止继续产生研究写入。

## 已确认需求

- 项目详情页标题右上角统一放置 `随机分组`、`确认分组`、`新增分组`、`项目完结`。
- 删除“占比与本轮随机结果仅在确认后保存。”文本说明。
- 继续遵守“占比与本轮随机结果仅在点击确认后保存”的业务规则。
- 新增 `项目完结` 按钮，完结后项目不能新增患者，也不能录入对应访视信息。
- 完结后 CRF 预览和 DOCX 导出仍允许。
- 患者详情页“直接确认入组到分组”入口中，不再显示已完结项目。
- 复用现有 `StudyProject.status = archived` 表示已完结，前端文案显示为“已完结”。
- 项目完结为单向操作，不提供恢复为进行中的入口。

## 非目标

- 不新增 `completed` 等项目状态字段，不做数据库状态迁移。
- 不改变 `ProjectPatient` 的已确认绑定语义。
- 不恢复批次、pending 入组、后端随机草稿或患者池实体。
- 不禁止已完结项目的只读查看、CRF 预览或 CRF 导出。
- 不在本次设计中新增审计日志、完结原因、完结时间字段或恢复流程。

## 页面交互设计

项目详情页标题右上角统一放置四个项目级动作，顺序为：

1. `随机分组`
2. `确认分组`
3. `新增分组`
4. `项目完结`

`随机分组` 与 `确认分组` 从 `ProjectGroupingBoard` 内部工具条上移到 `ProjectDetailPage` 标题右侧。`ProjectGroupingBoard` 仍负责患者选择、分组列、占比草案、本轮随机结果和拖拽调整，但动作触发由父页面统一承载。

`新增分组` 继续打开现有分组配置抽屉。新增成功后仍触发看板刷新与占比草案重平衡。

`项目完结` 是高风险动作，必须二次确认。确认弹窗应说明：

- 完结后不能新增或确认入组患者。
- 完结后不能新增、删除或修改分组配置，也不能保存分组占比。
- 完结后不能保存访视表单或标记访视完成。
- CRF 预览和 DOCX 导出仍可继续使用。
- 完结后不提供恢复入口。

项目已完结时，项目详情页展示“已完结”状态标签。`随机分组`、`确认分组`、`新增分组`、`项目完结` 不再可执行；看板进入只读模式，仍展示既有分组、已确认患者和历史数据。

删除看板内说明文案：

```text
占比与本轮随机结果仅在确认后保存。
```

## 后端规则设计

后端复用 `StudyProject.Status.ARCHIVED` 表示已完结。新增项目动作接口，例如：

```http
POST /api/studies/projects/{id}/complete/
```

接口行为：

- 若项目尚未完结，将 `status` 更新为 `archived`。
- 若项目已完结，返回成功并保持幂等，不重复修改数据。
- 不删除项目，不清理既有患者、访视、CRF 或处方数据。

所有会继续产生研究数据或修改项目配置的服务端入口都必须检查项目状态：

- `POST /api/studies/projects/{id}/confirm-grouping/`：已完结项目拒绝保存占比和确认入组。
- `POST/PATCH/PUT/DELETE /api/studies/groups/`：对应项目已完结时拒绝创建、修改或删除分组。
- `POST /api/patients/{id}/enroll-projects/`：提交项中包含已完结项目时拒绝直接确认入组。
- `POST /api/studies/project-patients/{id}/unbind/`：入组关系所属项目已完结时拒绝解绑，避免完结后改变项目样本。
- `PATCH/PUT /api/visits/{id}/`：访视所属项目已完结时拒绝保存表单、访视日期或状态变更。

只读入口继续允许：

- 项目列表、项目详情、分组列表、项目患者列表。
- 访视列表与访视详情。
- CRF `preview`。
- CRF `export`。

错误文案应清晰表达原因，例如：

```text
项目已完结，不能继续新增患者或录入访视。
```

## 前端数据流设计

`ProjectDetailPage` 拉取项目详情后，将 `project.status` 传给分组看板和分组配置组件。父页面统一持有动作按钮，并通过回调触发看板内的随机与确认逻辑。

`ProjectGroupingBoard` 增加只读模式：

- 患者勾选禁用。
- 本轮随机不可生成。
- 确认提交不可触发。
- 占比输入不可编辑。
- 本轮随机患者不可拖拽。
- 组删除入口隐藏或禁用。
- 已确认患者仍可查看详情，但解绑入口禁用或隐藏，避免完结后改变项目样本。

`ProjectGroupsTab` 在项目已完结时禁用新建分组入口，并阻止分组修改类动作。

患者详情页 `EnrollProjectsModal` 过滤掉 `status === "archived"` 的项目，因此完结项目不会出现在“直接确认入组到分组”的可选列表中。

研究录入页仍可展示完结项目的历史访视。进入 `VisitFormPage` 后，若访视所属项目已完结，页面展示只读提示并禁用所有输入、保存按钮和标记完成按钮。

CRF 预览页不因项目完结禁用选择、预览或导出。

## 错误处理

- 前端在已完结项目上禁用高风险写入按钮，减少误操作。
- 后端是最终边界，任何绕过前端的写请求仍必须被拒绝。
- 若用户停留页面期间项目被其他人完结，后端拒绝写入后，前端展示错误并刷新项目详情、分组、患者或访视数据。
- 项目完结动作成功后，前端刷新项目详情与项目列表，清空本轮随机结果和患者勾选状态。

## 测试与验收

前端测试：

- 项目详情页右上角按顺序展示 `随机分组`、`确认分组`、`新增分组`、`项目完结`。
- 看板内不再出现“占比与本轮随机结果仅在确认后保存。”。
- 点击 `项目完结` 需要二次确认，并调用项目完结接口。
- 项目已完结时，随机、确认、新增分组、占比输入、分组删除和本轮拖拽均不可用。
- 项目已完结时，看板仍展示既有分组和已确认患者。
- 患者详情直接入组弹窗不显示已完结项目。
- 访视表单在已完结项目下只读，保存和标记完成不可用。
- CRF 预览和 DOCX 导出按钮在已完结项目下仍可用。

后端测试：

- 项目完结接口将项目状态置为 `archived`。
- 已完结项目拒绝 `confirm-grouping` 保存占比或创建入组关系。
- 已完结项目拒绝通过患者详情直接确认入组。
- 已完结项目拒绝解绑既有入组患者。
- 已完结项目拒绝分组创建、修改和删除。
- 已完结项目拒绝访视 `PATCH/PUT`。
- 已完结项目仍允许 CRF `preview` 和 `export`。
- 已完结项目的只读列表和详情接口仍正常返回。

## 验收标准

- 用户在项目详情页右上角一眼看到分组随机、确认、新增分组和项目完结四个动作。
- 用户不再看到“占比与本轮随机结果仅在确认后保存。”说明文案。
- 未点击确认时，占比草案与本轮随机结果仍不会落库。
- 项目完结后不能再新增患者或修改分组配置。
- 项目完结后不能再录入或完成访视。
- 项目完结后仍能查看和导出 CRF。
- 患者详情页入组入口不会把已完结项目作为可选项。
