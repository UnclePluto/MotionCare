# 患者档案、研究项目与分组看板（管理端）设计

日期：2026-05-11  
范围：MotionCare 管理端（Django REST + React/Ant Design），聚焦患者域、研究项目列表/详情、分组看板与**分组确认**流程的交互与规则（**产品侧不暴露「批次」概念**，见下文「分组与随机」）。  
**已选交付策略**：纵向按域切片（患者 → 项目列表 → 项目详情看板），看板切片启动前对关键接口做一轮轻量契约对齐。

## 背景

当前能力缺口主要包括：患者档案缺少编辑与删除（及停用）闭环；患者缺少直接加入多个研究项目的入口；项目缺少删除能力；项目列表展示 CRF 模板列与入口分散；项目详情以多 Tab 承载分组、患者、随机分组与 CRF，与「以看板统一分配与草稿调整」的目标不一致。  
后端已具备部分 REST 能力（如 `PatientViewSet`、`StudyProjectViewSet` 等），本设计以前端与规则补齐为主，并按需增补接口与校验。

## 目标与非目标

### 目标（本期）

0. **删除类操作（全局交互，强制）**  
   - 界面中凡 **删除、解绑、永久移除** 等可导致 **数据不可恢复** 或 **关联数据被删除/作废/失效** 的动作（含但不限于：患者**物理删除**、**研究项目删除**、看板 **「从本项目移除」解绑**、看板/草案中 **「确认删除」** 且会触发服务端删除或等价终止——**具体清单由实现计划枚举并与本段对齐**），均须 **二次确认**（Modal 或等效阻断式交互，**禁止单步无确认直删**）。  
   - 二次确认界面 **必须** 展示 **关联信息删除/作废警告说明**：根据当前对象 **动态汇总** 将影响的数据类别与范围（示例：入组关系、访视与 CRF、处方、训练记录、导出文件、分组状态等）；若经预检 **无** 下游关联，亦须给出 **明确后果文案**（如「将永久删除该档案且不可恢复」），**不得空白确认或仅展示通用标题**。  
   - 若业务规则 **禁止** 删除：用户点击后仍须进入 **带说明的阻断流程**（Modal 或同等级组件），展示 **禁止原因** 与 **已识别关联项**；不得仅依赖一行 toast 作为唯一反馈（可与预检接口配合）。  
   - **停用**（如将患者 `is_active` 置为 false、或未来等价的「档案停用」状态）：**同样需要二次确认**；确认界面须说明 **业务影响**（例如：默认列表是否隐藏该档案、停用期间是否禁止加入新项目/开具处方等——以实现在字段与权限上的行为为准，**须在实现计划中写死并与验收对齐**）。

1. **患者档案**  
   - 列表行内与详情页均可进入**编辑**。  
   - **删除/停用等高风险操作**仅在患者详情提供。  
   - **物理删除**须遵守上文 **「删除类操作（全局）」**；确认框内须按预检结果展示关联项目、CRF/访视、处方等 **警告说明**（以实际可查询关联为准）。  
   - **删除规则**：若仍存在项目关联（`ProjectPatient`）或按产品策略认定的 CRF 相关数据，则**禁止物理删除**；引导停用或先解除全部项目关联。  
   - **加入研究项目**：支持一次**多选多个项目**；选择器中**已在该项目下的记录置灰且不可勾选**；展示提示：加入后请到**各项目详情看板**完成勾选与随机分组（各项目独立；具体比例以看板 `target_ratio` 为准）。  
   - **患者详情**：展示该患者参与的研究项目列表；**每一行提供「打开 CRF」**（对应该 `ProjectPatient`，与 `/crf?projectPatientId=` 一致）。

2. **研究项目列表**  
   - **隐藏**「CRF 模板」列（不在列表暴露模板版本）。  
   - 提供 **「详情」** 与 **「删除」**；须遵守上文 **「删除类操作（全局）」**（二次确认 + 关联影响说明；若禁止删除则展示原因与关联项）。  
   - **删除规则**：若存在任意 `ProjectPatient`，或存在**服务端未闭环的待确认分组工作集**（实现可与 `ProjectPatient.grouping_status` / 临时 staging 对齐，**不再以「分组批次」对用户表述**），则**禁止删除**并返回明确错误文案。

3. **项目详情（看板化）**  
   - **分组与随机（无「批次」产品语义，2026-05-11 修订）**  
     1. **草稿**：随机结果与列间拖拽的调整，**优先视为前端本地草稿**即可；不要求持久化多份历史草稿，**不要求记住或展示「第几次批次」**。若实现选择写入服务端以便多端协作，对用户仍只呈现「当前待确认草案」，不展示批次编号或批次列表。  
     2. **确认后锁定**：某 `ProjectPatient` 一旦完成**分组确认**，**禁止跨列拖拽改组**；仅允许**将该患者从本研究项目解绑**（删除该 `ProjectPatient` 记录，即退出本项目），且**必须二次确认**（Modal）。二次确认文案须提示：若已存在 **与本项目、该入组关系相关的 CRF 数据**，解绑后将 **标记删除或作废**；**处方将标记无效**，且 **医生端不再展示**。  
     3. **已确认展示**：已确认分组的患者仍显示在其所属列中，卡片**整体置灰**（与待确认可拖拽卡片区分）。  
     4. **随机分组范围**：每次触发「随机分组」时，**参与集合 = 当前已勾选 ∩ 分组状态仍为待确认（未确认）**的患者；**已确认者不参与再随机**。用户可通过 **「取消随机」** 放弃本次未确认的随机草案（勾选集合与列上草案回到规则定义的前一状态），或通过 **「确认删除」** 将某患者从当前勾选或草案上下文中移除，使其不再进入后续随机；**「确认删除」** 仍须遵守 **「删除类操作（全局）」**（二次确认 + 若涉及服务端删除或状态终止，须含关联影响说明；若仅本地勾选变化，确认框须至少说明 **对当前草案的后果**）。  
     5. **解绑时的 CRF 与处方（数据治理）**：解绑执行后，凡 **已存在且对应该 `ProjectPatient`（该患者在本项目下的入组关系）的 CRF 实质数据**，须 **全部标记删除或作废**（审计优先时采用「作废」语义，不得再作为有效 CRF 参与业务；实现可用软删/作废状态位，细则见实现计划）。与此入组关系相关的 **全部处方** 须 **标记无效**（与现有处方状态机兼容时，可归并为「已终止」或单独增加「无效/作废」状态，由实现计划选定），**医生端所有默认列表与工作台均不得展示**（含处方列表、开具与执行入口等；若需管理端只读审计，单列权限与入口）。  
   - **患者池**：面向**全库患者**；以「尚未加入本项目」为主要操作对象；**已加入本项目**者在池中**弱化显示或过滤**（实现可选，默认与「不可重复勾选」一致）。  
   - **列**：每组一列（竖队列）；列头为**组名**与**占比**；多列占比**展示合计 100%**，并与后端 **`target_ratio` 整数权重**双向一致（见「占比与尾差」）。  
   - **列内卡片**：简略信息含**姓名、性别、手机尾号**；**仅待确认**者支持**跨列拖拽**调整草案分组；悬浮露出**患者详情**与**从当前勾选/草案移除**等操作（与上条「确认删除」一致，具体动词与 API 在实现计划中落到端点）。  
   - **原项目详情中的 CRF 主入口**：不再作为 CRF 主路径（与「CRF 自患者侧进入」一致）；可移除原 CRF Tab 或降级为次要链接，**默认移除主 Tab**，避免与信息架构冲突。

**实现备注（给工程，非产品文案）**：当前代码存在 `GroupingBatch` 与 `ProjectPatient.grouping_batch`；实现本规格时可收敛为「单项目至多一个待确认工作集」或逐步改为仅依赖 `ProjectPatient` 状态字段承载确认语义，**但 UI 与文档不再出现「批次」用语**。迁移与兼容在实现计划中单独列出。  
**CRF 落库现状**：CRF 预览/导出主要聚合自 `VisitRecord`（及患者基础信息），另有 `CrfExport`；解绑时若外键为 `CASCADE` 会物理级联删除，与产品「标记删除/作废」不一致时，须在实现计划中改为 **先批量作废/软删再解除关联**，或调整 `on_delete` 与影子/归档策略。处方模型已有 `TERMINATED` 等状态，「无效」是否复用或扩展在实现计划中写死，并统一 **医生端 API 查询默认过滤**。

4. **占比与尾差**  
   - **不改库表语义**：持久化仍为各组 `target_ratio`（正整数）。  
   - 列头以权重换算为**整数百分比**展示，须处理**舍入尾差**（例：权重 1:1:1 → 展示 **33%、33%、33%**，不展示无限小数）。  
   - 用户在看板修改占比时，前端换算回一组整数权重，**PATCH** 更新各组；须定义**溢出保护**与写回后仍满足随机算法输入约束（细则在实现阶段用单测锁死）。

5. **随访与 CRF**  
   - **本期不接**「从随访流程进入 CRF」的页面与路由。  
   - 在患者详情等位置可放置**简短占位说明**（例如未来将在此关联访视与 CRF）。

### 非目标（本期不做）

- 随访/访视 UI 与 CRF 的深度串联。  
- 移动端看板专项优化（可按桌面优先实现，窄屏降级策略另议）。  
- 与分组无关的报表、权限模型大改（沿用现有认证与视图集前提，除非实现时发现硬缺口）。

## 方案选择（已确认）

采用 **纵向按域切片** 为主：

1. **切片一**：患者（编辑、详情侧删除/停用规则、多项目加入、详情项目表 + 每行 CRF）。  
2. **切片二**：项目列表（隐藏 CRF 列、详情与删除及后端校验对齐）。  
3. **切片三**：项目详情看板（池、勾选、列头占比、拖拽草案、随机、确认/取消随机）；**启动前**对池查询、比例 PATCH、批量入组、**确认写入**等做**轻量契约对齐**（小范围 API/序列化器评审即可；**去除对用户可见的「批次」API 形态**）。

## 关键决策汇总

| 主题 | 决策 |
|------|------|
| 删除类交互 | **一律二次确认**；确认界面 **必须含关联删除/作废警告说明**（动态或兜底后果文案）；禁止删除时须 **阻断式说明** |
| 停用 | **同样需要二次确认**；须含 **业务影响说明**（列表、入组、处方等与实现写死的行为） |
| 患者物理删除 | 存在项目关联或认定 CRF 数据则禁止；否则允许；提供停用路径；交互仍须满足上条 |
| 项目删除 | 有项目患者或未闭环待确认分组工作集则禁止；交互仍须满足上条 |
| 分组草案与确认 | **无批次产品概念**；草案可仅前端；确认后锁定、列内置灰；随机范围见正文第 3 条 |
| 已确认者「从组内删除」 | **直接解绑项目**（删除 `ProjectPatient`），二次确认；不解绑则不可改组 |
| 解绑后 CRF / 处方 | 已有 CRF 实质数据 **标记删除或作废**；处方 **标记无效**；**医生端不展示**（管理端审计另议） |
| 患者池范围 | 全库；已入本项目者弱化或过滤 |
| 占比存储 | 仍为 `target_ratio`；列头为展示换算 + 尾差规则 |
| 批量加入项目 | 多选；已加入项目在选择器中置灰不可选；提示到各项目看板完成勾选与随机分组 |
| CRF 主路径 | 患者详情按项目分行 + 每行打开 CRF；项目详情弱化/移除 CRF 主 Tab |
| 编辑入口 | 列表与详情均可编辑；删除仅详情 |
| 随访 | 下一迭代；本期占位文案 |
| PostgreSQL 与锁 | 确认写入路径须事务安全；若沿用现有 confirm 实现，延续已修复的 `select_for_update(of=("self",))`，避免可空外键与 `select_related` 组合导致的外连接加锁错误 |

## 信息架构与路由（概念）

- **患者列表**：行内「编辑」「详情」等；「加入项目」可放列表或详情（实现计划定具体位置，须满足多选项目交互）。  
- **患者详情**：编辑、删除/停用、加入项目、参与项目表（每行 CRF）、随访占位说明。  
- **项目列表**：列含项目名称、状态、患者数（若后端已有或后续补全）等；**不含** CRF 模板列；操作「详情」「删除」。  
- **项目详情**：单页看板为主；分组「元数据」配置（组名、启用、排序等）通过**抽屉或次级面板**保持可达，不丢失现有 `StudyGroup` 管理能力。

## 看板交互（概念级）

1. **患者池区域**：搜索/筛选；多选患者 → 进入「当前勾选集合」（可配合本地草稿恢复）。  
2. **分组列区域**：每列对应 `StudyGroup`；列头编辑占比（联动 100% 展示）；列内为卡片列表；**仅未确认**者支持拖拽跨列。  
3. **卡片操作**：进入患者详情路由；**已确认**者置灰、不可拖拽；**未确认**者可从勾选/草案移除（与正文「确认删除」一致，仅影响勾选/草案，不必然解绑）。**已确认**者仅允许 **「从本项目移除」**（解绑 = 删除 `ProjectPatient`），须二次确认。  
4. **工具条**：提供「随机分组」「取消随机」「确认分组」等；语义与正文「分组与随机」四条一致。

## 错误处理与幂等

- **批量加入**：前端置灰降低冲突；服务端仍依赖 `(project, patient)` 唯一约束，冲突时返回可读错误，避免静默数据异常。  
- **删除患者/项目/解绑**：API 返回明确 HTTP 状态与 `detail`；前端须与 **「删除类操作（全局）」** 一致：成功路径前已二次确认并展示关联警告；失败路径以 **Modal 级说明** 呈现后端原因与已识别关联（不得仅 toast）。  
- **确认写入**：保持事务与行级锁策略与已修复实现一致（若后端仍包裹为单次 confirm），避免 `FOR UPDATE` 作用于外连接可空侧。  
- **解绑事务**：作废 CRF 相关数据、将处方置无效与删除/归档 `ProjectPatient` 宜同事务提交；失败则整体回滚并返回可读错误。

## 测试与验收要点

- 患者：编辑保存、禁止删除场景、多选加入与置灰、详情多行 CRF 跳转；**物理删除**二次确认 + 关联警告（允许删除时）或阻断说明（禁止时）。  
- 项目：列表无 CRF 列、**删除**二次确认 + 关联警告、禁止删除场景的阻断说明。  
- 看板：占比展示尾差、写回 `target_ratio`、随机范围（勾选∩未确认）、取消随机、拖拽草案、确认后不可拖、已确认置灰、解绑/草案确认删除等 **均符合全局二次确认 + 关联警告**、解绑后 CRF 作废与处方无效、医生端列表过滤、确认写入。  
- 后端单测：删除保护、比例换算与舍入、确认路径锁与查询（若 API 从「批次」收敛，单测同步改名）。
