# 项目详情分组看板占比草案与拖拽调整设计

> 状态：review
> 日期：2026-05-13
> 范围：项目详情分组看板的占比编辑、确认提交、本轮随机拖拽调整、组列与患者卡片 UI。
> 关联：`docs/superpowers/specs/2026-05-12-frontend-only-randomization-confirmed-binding-design.md`、`docs/superpowers/specs/2026-05-13-grouping-board-selection-polish-design.md`
> 实施基线 commit：66089ac

## 背景

项目详情页已经回归为单一分组看板，并且随机分组已收敛为纯前端临时结果：刷新或切换项目即丢失，只有点击确认后才创建 `ProjectPatient`。现有看板仍暴露了“权重”相关交互：组列占比由后端 `target_ratio` 权重换算展示，页面提供“应用占比到权重”按钮，分组配置抽屉中新建分组也要求填写 `target_ratio`。

本次迭代要把产品语义改成“占比”，取消权重概念，并让占比修改和本轮随机患者一样成为当前页面草案：只有点击“确认分组”时才落库。

## 已确认需求

- 取消权重逻辑和所有面向用户的权重文案；创建分组不再要求填写权重或比例。
- 分组随机比例直接由用户填写的占比决定，启用组占比合计必须为 100%。
- 新增一个启用组后，前端自动把所有启用组占比调整为均衡分布，例如 3 组为 `34/33/33`。
- “确认分组”按钮移动到“随机分组”旁边，并通过颜色区分；取消独立“应用占比到权重”按钮。
- 组患者列表要更紧凑，尽量一到两行展示患者信息与操作。
- 本轮随机生成后，允许拖拽本轮随机患者调整所在分组。
- 分组删除按钮改为组卡片内部右上角圆形 X，鼠标悬浮到组卡片时显示，不能越界导致显示不全。
- 允许没有本轮随机患者时点击“确认分组”，只保存当前占比草案。

## 非目标

- 不恢复 `GroupingBatch`、pending 入组、后端随机草稿或患者池实体。
- 不让已确认患者拖拽改组；已确认患者仍只能通过解绑后重新随机/确认进入其他组。
- 不做多端实时协作或草案持久化；页面刷新、切换项目或关闭浏览器仍丢弃未确认草案。
- 不把分组删除改成无确认操作；删除仍沿用高风险操作二次确认。
- 不在本次改造中新增审计日志、版本历史或比例修改记录。

## 产品语义

`StudyGroup.target_ratio` 在 API 字段名上可继续沿用，但产品语义改为“占比整数”。确认后的已保存启用组配置中，`target_ratio` 应表示百分比，所有启用组总和必须为 100。停用组不参与占比合计和随机，但如果已有已确认患者仍可在看板中作为历史列展示。

看板中的占比编辑是本地草案。用户改动占比、新增分组触发均衡、删除分组后重新均衡，都只改变前端状态，不立即 PATCH 后端。点击“确认分组”后，当前占比草案才和本轮随机结果一起提交。确认接口是前端保存比例的唯一入口，常规看板交互不再直接 PATCH `StudyGroup.target_ratio`。

“确认分组”的含义扩展为“确认当前看板草案”：

- 有本轮随机患者时：保存占比，并按最终分组确认患者入组。
- 没有本轮随机患者时：只保存占比。

## 前端交互设计

### 顶部操作区

看板保留一条简洁操作区：

- “随机分组”：主色按钮，使用当前勾选患者和当前占比草案生成本轮随机结果。
- “确认分组”：成功色按钮，紧邻“随机分组”，提交当前占比草案和本轮随机结果。
- 辅助说明收敛为一条短文案：占比与本轮随机结果仅在确认后保存。

删除“应用占比到权重”按钮，以及“权重”“应用到权重”等相关文案。

### 分组占比

每个启用组列头展示可编辑占比输入框，单位为 `%`。输入值必须是整数，建议范围为 1 到 100。看板在本地计算启用组占比总和：

- 合计为 100：允许随机和确认。
- 合计不为 100：允许继续编辑，但确认时阻止提交，并提示当前合计与差值。
- 任一启用组占比小于等于 0：阻止随机和确认。

随机分组直接使用当前占比草案计算人数分配，不依赖后端已保存比例。

### 新增分组均衡

分组配置抽屉中新建分组只要求输入分组名称。分组实体可以先创建到后端；若后端字段仍要求 `target_ratio`，可使用系统默认占位值，但该值不视为最终项目占比。新分组创建成功后，前端基于最新启用组列表自动生成均衡占比草案：

- 2 组：`50/50`
- 3 组：`34/33/33`
- 4 组：`25/25/25/25`
- 其他数量：用整数均分，余数从排序靠前的组开始依次 +1，确保合计严格为 100。

这次均衡不立即写回后端。用户可继续手动调整，最终点击“确认分组”才保存。若用户在新增组后未确认即刷新页面，页面可根据当前启用组重新初始化占比草案；最终仍以确认接口保存的 100% 配置为准。

### 删除分组

分组删除入口改为圆形 X：

- 位置：组卡片内部右上角。
- 显示：默认隐藏，鼠标悬浮到组卡片时显示。
- 尺寸：圆形小按钮，使用危险色，但不占用常规标题布局。
- 防遮挡：组标题区域右侧预留按钮空间，X 不外飘到卡片边界外，避免被父容器或横向滚动区域裁剪。

点击后仍打开二次确认弹窗。删除成功后，前端对剩余启用组重新均衡占比，作为本地草案等待确认落库。若后端因患者关联等原因拒绝删除，展示后端错误，不改动本地组列表。

### 患者卡片密度

组内患者卡片改为紧凑布局：

- 首行优先展示姓名、性别、手机号尾号、状态标签。
- 操作文案精简：`详情`、`移除`、`解绑`。
- 本轮随机患者展示“本轮”或“本次随机”标签。
- 已确认患者继续置灰展示，并展示“已确认”标签。
- 组列宽度比当前实现更宽，减少患者信息换行；多列时继续横向滚动。

目标是在常见患者姓名长度下，一张患者卡片一到两行内展示完主要信息与操作。

### 本轮随机拖拽

本轮随机患者支持在启用组之间拖拽调整：

- 仅 `localAssignments` 中的本轮随机患者可拖拽。
- 已确认患者不可拖拽。
- 拖拽后只修改本地 `patientId -> groupId` 映射。
- 点击确认时，按拖拽后的最终分组提交 `assignments`。
- 再次点击“随机分组”会按当前勾选患者和当前占比草案重新生成本轮结果，并覆盖拖拽调整。

若患者从全量患者区域取消勾选，仍沿用现有设计：同步移除该患者的本轮随机结果。

## 数据流与接口

前端维护三类状态：

- `percentByGroupId`：当前启用组占比草案。
- `poolSelected`：本次参与随机的患者选择。
- `localAssignments`：本轮随机患者及其当前组，可由随机生成或拖拽修改。

推荐扩展 `POST /api/studies/projects/{id}/confirm-grouping/`：

```json
{
  "group_ratios": [
    { "group_id": 10, "target_ratio": 34 },
    { "group_id": 11, "target_ratio": 33 },
    { "group_id": 12, "target_ratio": 33 }
  ],
  "assignments": [
    { "patient_id": 1, "group_id": 10 },
    { "patient_id": 2, "group_id": 11 }
  ]
}
```

`assignments` 允许为空或缺省，但 `group_ratios` 至少应包含当前项目全部启用组的占比。后端在同一事务中完成：

1. 校验所有 `group_ratios.group_id` 属于当前项目且为启用组。
2. 校验提交的启用组集合与当前项目启用组一致，避免漏提交导致合计不可信。
3. 校验 `target_ratio` 为正整数，且合计严格为 100。
4. 更新各启用组 `target_ratio`。
5. 若存在 `assignments`，沿用现有确认入组校验：患者存在、分组属于当前项目且启用、同一请求无重复患者、患者尚未在当前项目确认入组。
6. 创建 `ProjectPatient` 并生成默认访视。

任一校验失败时整体回滚，不保存部分比例，也不创建部分入组关系。

## 错误处理

- 前端发现占比合计不是 100 时，不提交后端，提示“启用组占比合计须为 100%，当前为 X%”。
- 前端发现启用组占比小于等于 0 时，不允许随机或确认。
- 后端发现分组被删除、停用、跨项目或启用组集合变化时返回明确错误；前端刷新分组与患者数据，并清空本轮随机结果。
- 后端发现患者已确认入组、重复提交或患者不存在时拒绝确认；前端展示错误并刷新患者数据。
- 只保存占比成功时提示“占比已保存”。
- 保存占比且确认入组成功时提示“分组已确认”。

## 测试与验收

前端工具函数测试：

- 均衡占比函数对 2、3、4、5 组输出整数且合计 100。
- 3 组输出 `34/33/33`。
- 占比校验能识别合计不是 100、含 0 或负数的情况。
- 随机分组使用占比草案而非后端旧比例。

前端组件测试：

- 看板不再出现“权重”与“应用占比到权重”。
- “确认分组”与“随机分组”相邻展示，且确认按钮在无本轮随机患者但占比可保存时可用。
- 无本轮随机患者时点击确认，会提交 `group_ratios` 且 `assignments` 为空或缺省。
- 占比合计不是 100 时点击确认不会请求后端。
- 新增分组成功后，启用组占比草案自动均衡。
- 本轮随机患者可拖拽到另一组，确认提交时使用最终组。
- 已确认患者仍禁选、置灰、不可拖拽，解绑入口仍走二次确认。
- 分组删除 X 默认隐藏，hover 组卡片后出现，且不位于卡片边界外。

后端测试：

- `confirm-grouping` 可只提交 `group_ratios` 并更新占比。
- `confirm-grouping` 可同时提交 `group_ratios` 与 `assignments`，在同一事务内保存比例并确认入组。
- 占比合计不是 100、存在 0 或负数、漏提交启用组、提交停用组、提交跨项目分组时拒绝并不落库。
- assignments 中重复患者、未知患者、已确认入组患者、未知分组、停用分组仍拒绝。
- 创建入组关系后仍生成默认访视。

## 验收标准

- 用户在项目详情中看不到“权重”相关文案或按钮。
- 用户新增分组时只填写名称；新增成功后看板占比自动均衡。
- 用户调整占比后刷新页面前未点击确认，则数据库比例不变；点击确认后才保存。
- 用户可以只保存占比，不需要先随机患者。
- 用户可以对本轮随机患者拖拽换组，确认后按最终组落库。
- 分组删除 X 在 hover 时显示完整，不被容器裁剪。
- 现有纯前端随机、确认后绑定、已确认患者禁选与解绑规则保持不变。
