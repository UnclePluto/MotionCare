# 医生管理与默认密码强制修改设计

## 背景

MotionCare 当前已使用 `accounts.User` 作为后台账号模型，角色包含 `super_admin`、`admin`、`doctor`。这一版不做细粒度角色隔离，所有医生默认具备 admin 级使用权限；医生管理入口、医生列表、新建医生、编辑医生基础资料均对已登录后台用户开放。

本设计只覆盖医生账号管理与当前登录账号改密，不引入独立 `Doctor` 模型，不做审计日志、账号删除、账号停用、管理员重置他人密码或复杂权限策略。

## 目标

1. 侧边栏新增“医生管理”入口。
2. 医生管理列表展示医生姓名、脱敏手机号、创建时间和编辑按钮。
3. 列表页提供“添加医生”按钮，进入独立新建页面。
4. 新建医生填写姓名、性别、手机号；手机号需校验合规。
5. 新建成功后使用默认密码 `888888`，提示“创建成功。默认密码为 888888，建议登录后立刻修改。”
6. 右上角当前医生姓名可点击进入“我的账号”，可编辑当前登录账号基础资料并修改密码。
7. 使用默认密码创建的账号登录后必须先修改密码；弹窗不可关闭、不可跳过，改密成功后才可继续使用系统。

## 后端设计

### 数据模型

沿用 `accounts.User`，新增字段：

- `gender`：枚举字段，取值 `male`、`female`、`unknown`，默认 `unknown`。
- `must_change_password`：布尔字段，默认 `false`。

`date_joined` 作为医生账号创建时间展示，不额外增加 `created_at` 字段。

### 权限与范围

这一版所有已登录后台用户都可以使用医生管理能力，相当于医生默认拥有 admin 权限。实现上沿用现有 `IsAdminOrDoctor` 登录与角色检查，不新增角色隔离。

为保持 `/api/me/` 返回的权限点与 UI 能力一致，`doctor` 角色的权限列表应包含 `user.manage`。

医生管理列表默认只展示 `role=doctor` 的账号。`admin` 与 `super_admin` 账号不进入“医生管理”列表。

### API 行为

`GET /api/me/` 返回字段增加：

- `gender`
- `must_change_password`

`/api/accounts/users/` 继续作为医生账号资源接口，但序列化与写入行为收紧：

- 列表字段：`id`、`name`、`gender`、`phone`、`role`、`date_joined`、`must_change_password`、`is_active`。
- 新建医生：只创建 `role=doctor`，忽略客户端传入的其他 role；默认密码固定为 `888888`；`must_change_password=true`。
- 编辑医生基础资料：允许更新姓名、性别、手机号；不允许通过该接口修改密码。
- 手机号必须唯一，格式按中国大陆 11 位手机号校验，采用规则 `^1[3-9]\d{9}$`。

新增当前账号改密接口：

- `POST /api/accounts/users/me/change-password/`
- 请求字段：`old_password`、`new_password`、`confirm_password`
- 校验旧密码正确。
- 校验新密码与确认密码一致。
- 新密码不能为 `888888`。
- 成功后更新密码并设置 `must_change_password=false`。

当前账号基础资料可通过 `PATCH /api/accounts/users/{me.id}/` 更新，保存后前端刷新 `/api/me/`。

### 强制改密硬拦截

如果当前用户 `must_change_password=true`，除以下接口外，其他需要登录的 API 返回 403：

- `GET /api/me/`
- `POST /api/auth/logout/`
- `POST /api/accounts/users/me/change-password/`

返回提示应明确为“请先修改默认密码”。这样即使用户绕过前端弹窗，也不能继续调用业务接口。

## 前端设计

### 路由与入口

新增路由：

- `/doctors`：医生管理列表页。
- `/doctors/new`：添加医生页。
- `/doctors/:doctorId/edit`：编辑医生基础资料页。
- `/account`：我的账号页。

侧边栏新增“医生管理”入口，所有已登录用户可见。Header 右上角当前姓名与脱敏手机号变成可点击入口，点击跳转 `/account`；退出按钮保留。

### 医生管理列表

列表标题为“医生管理”，右上角按钮为“添加医生”。

表格列：

- 医生姓名
- 脱敏手机号，复用患者列表的手机号脱敏规则
- 创建时间，来自 `date_joined`
- 操作：编辑

点击“编辑”进入 `/doctors/:doctorId/edit`。列表编辑只维护基础资料，不提供删除、停用或改他人密码。

### 添加医生页

独立页面 `/doctors/new`，字段：

- 姓名，必填，提交时 trim。
- 性别，单选：男、女、未知；默认未知。
- 手机号，必填，前端按 11 位中国大陆手机号校验。

创建成功后提示：“创建成功。默认密码为 888888，建议登录后立刻修改。”然后返回 `/doctors`。

### 编辑医生页

独立页面 `/doctors/:doctorId/edit`，字段：

- 姓名
- 性别
- 手机号

保存成功后提示成功并返回 `/doctors`。手机号变更后医生列表缓存失效并刷新。

### 我的账号页

`/account` 展示并允许编辑当前登录账号基础资料：

- 姓名
- 性别
- 手机号

同页提供“修改密码”区域：

- 原密码
- 新密码
- 确认新密码

改密成功后提示成功并刷新 `/api/me/`，确保 Header 和 `must_change_password` 状态立即更新。

### 强制改密弹窗

前端在登录态加载完成后读取 `me.must_change_password`。当值为 `true` 时显示不可关闭 Modal：

- 不允许点击遮罩关闭。
- 不显示关闭按钮。
- 不允许跳过或“稍后再说”。
- 只提供修改密码表单和退出登录按钮。

改密成功前用户不能继续操作系统页面。改密成功后刷新 `/api/me/`，弹窗消失。

## 异常处理

- 手机号格式错误：提示“请输入 11 位有效手机号”。
- 手机号重复：提示“该手机号已存在”。
- 改密原密码错误：提示“原密码错误”。
- 新密码与确认密码不一致：前端先拦截，后端也校验。
- 新密码为 `888888`：提示“新密码不能使用默认密码”。
- 强制改密拦截：403，提示“请先修改默认密码”。

## 测试计划

### 后端

- `User` 新字段默认值：`gender=unknown`、`must_change_password=false`。
- 新建医生接口创建 `role=doctor`，默认密码 `888888` 可登录，且 `must_change_password=true`。
- `/api/me/` 返回 `gender` 与 `must_change_password`。
- 医生列表只返回 `role=doctor` 账号，包含 `date_joined`。
- 手机号格式与唯一性校验生效。
- 强制改密状态下普通业务接口返回 403。
- 强制改密状态下 `/api/me/`、退出登录、改密接口可用。
- 改密成功后 `must_change_password=false`，旧密码不可登录，新密码可登录。
- 新密码为 `888888` 时拒绝保存。

### 前端

- 菜单展示“医生管理”入口。
- 医生列表展示姓名、脱敏手机号、创建时间和编辑按钮。
- “添加医生”跳转独立新建页。
- 新建医生成功后显示默认密码提示并返回列表。
- 编辑医生基础资料成功后返回列表并刷新数据。
- Header 当前姓名入口跳转 `/account`。
- 我的账号页可更新基础资料并刷新 `/api/me/`。
- 修改密码成功后刷新登录态。
- `must_change_password=true` 时弹窗不可关闭，改密成功前不可继续使用页面。

## 非目标

- 不新增独立 `Doctor` 模型。
- 不做账号删除、账号停用、管理员重置他人密码。
- 不做完整审计日志。
- 不做细粒度权限配置；这一版所有医生默认 admin 权限。
- 不修改患者、项目、随机分组、处方、CRF 等既有业务模型决策。
