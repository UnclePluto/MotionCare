# 运动训练处方与患者端模拟跟练设计

## 背景

MotionCare 既有 PRD 已确认处方是项目患者维度下的版本化训练计划：一个 `ProjectPatient` 同一时刻只有一条生效处方，调整处方会生成新版本，训练记录只能基于当前生效处方录入。

本次处方模块聚焦 `docs/other/课题任务.xlsx` 中的“运动训练”。该表提供了动作类型、动作名称、具体执行、动作要点、建议频次/时长和 AI 监督信息。第一版需要让医生在后台给项目患者开具动作类处方，并提供一个 Web 患者端模拟跟练页，方便测试“查看当前处方动作 -> 看视频跟练 -> 提交训练记录”的完整链路。

## 已确认决策

1. **采用医生端处方 + Web 患者端模拟跟练方案**：不做真实患者账号或小程序登录，但页面与 API 语义按未来患者端设计。
2. **动作库固定**：医生只能查看和选择系统预置动作，不能新增、编辑或删除动作。后续动作变更通过系统版本迭代完成。
3. **视频资源通过 URL 配置**：动作库与处方动作快照预留 `video_url`；测试环境可先使用占位或测试视频地址。
4. **处方动作参数按动作类型切换样式**：每个动作独立设置处方参数；表单根据 `action_type` 区分“按持续时间计算”和“按个数/次数计算”。
5. **前端不暴露草稿流程**：医生提交处方后直接生成新版本并立即生效；旧 active 处方自动归档。
6. **患者一次跟练一个动作**：每次提交生成一条 `TrainingRecord`。
7. **患者端模拟页只显示当前生效处方动作**：没有处方时不展示动作库动作。
8. **动作说明聚合存储**：Excel 中的具体执行和动作要点合并为一个动作说明文案字段，处方快照也保存该聚合文案。
9. **AI 监督仅保存是否支持**：本期不保存 AI 监督详情，只保存动作是否具备 AI 监督能力。

## 目标

- 预置 `课题任务.xlsx` 中 5 个运动训练动作，作为固定动作库。
- 医生端可只读查看固定动作库及视频配置状态。
- 医生可为项目患者开具或调整处方，选择固定动作并逐动作设置参数。
- 新处方提交后立即生效，并归档同一项目患者旧生效处方。
- 处方动作保存动作库关键字段快照，动作库后续变更不影响历史处方解释。
- 提供 Web 患者端模拟跟练页，读取当前生效处方，按单个动作播放视频并提交训练记录。
- 后端保证训练提交只能基于当前 active 处方及其动作。

## 非目标

- 不做真实患者账号、微信小程序、短信登录或一次性外部链接。
- 不做视频上传、对象存储管理、防盗链或签名 URL。
- 不做 AI 动作识别、自动评分、摄像头采集或动作实时监督。
- 不开放医生维护动作库能力。
- 不支持按旧处方版本补录历史训练。
- 不实现处方计划周期、自动结束日期或提醒推送。

## 固定动作库

本期从 `课题任务.xlsx` 预置“运动训练”下的 5 个动作：

| 编码建议 | 动作类型 | 动作名称 | 建议频次/时长 | 备注 |
| --- | --- | --- | --- | --- |
| `motion-aerobic-high-knee` | 有氧训练 | 椰林步道模拟（原地高抬腿+摆臂） | 3 次/周，20 分钟/次 | 原地高抬腿、摆臂。 |
| `motion-balance-sit-stand` | 平衡训练 | 坐站转移训练 | 2 次/周，15 分钟/次 | 椅子坐站转换。 |
| `motion-resistance-row` | 抗阻训练（弹力带） | 坐姿划船 | 2 次/周，10 分钟/次 | 弹力带坐姿背部训练。 |
| `motion-resistance-leg-kickback` | 抗阻训练（弹力带） | 腿部后踢 | 2 次/周，10 分钟/次 | 弹力带下肢后踢。 |
| `motion-resistance-shoulder-press` | 抗阻训练（弹力带） | 肩部推举 | 2 次/周，10 分钟/次 | 弹力带肩部推举。 |

动作库字段初版：

- `source_key`：系统内稳定编码，用于 seed 幂等更新。
- `name`：动作名称。
- `training_type`：固定为 `运动训练`。
- `internal_type`：固定为 `motion`。
- `action_type`：如有氧训练、平衡训练、抗阻训练；同时驱动处方表单的参数样式。
- `instruction_text`：动作说明文案，聚合 Excel 中的具体执行和动作要点。
- `suggested_frequency`：来自 Excel 的建议频次文本。
- `suggested_duration_minutes`：建议单次时长。
- `suggested_sets`：建议组数，Excel 未明确时可为空。
- `suggested_repetitions`：建议次数，Excel 未明确时可为空。
- `default_difficulty`：默认难度，Excel 未明确时可为空。
- `video_url`：动作视频 URL，可为空。
- `has_ai_supervision`：是否支持 AI 监督。
- `is_active`：是否启用；停用动作不可新开处方，但历史快照仍可展示。

本期处方参数样式按 `action_type` 映射：

- 有氧训练：时长型，处方表单重点填写每周频次和单次时长。
- 平衡训练：计数型，处方表单重点填写每周频次、组数和次数；单次时长可作为辅助建议值。
- 抗阻训练：计数型，处方表单重点填写每周频次、组数和次数；单次时长可作为辅助建议值。

实施前复核点：`课题任务.xlsx` 中“动作类型”列存在两个同名表头，且频次/时长位于合并单元格后的动作要点列中。实施计划第一步必须对照 Excel 与用户复核动作字段映射，确认哪些内容进入动作库字段，哪些内容只作为展示说明。

## 处方模型与规则

现有 `Prescription` 继续作为版本化处方主体，属于 `ProjectPatient`。前端不暴露草稿，医生每次提交都创建下一版本并立即生效。

创建或调整处方的服务层规则：

1. 后端在事务中读取项目患者当前 active 处方。
2. 计算下一版本号，创建新 `Prescription`。
3. 为每个被选动作创建 `PrescriptionAction` 快照。
4. 将新处方置为 `active`，`effective_at` 为当前时间。
5. 将同一项目患者旧 active 处方置为 `archived`。
6. 如果提交时当前 active 版本与前端基线版本不一致，拒绝提交并提示刷新，避免并发覆盖。

`PrescriptionAction` 快照需要覆盖：

- 动作库引用 `action_library_item`
- 动作名称快照
- 训练类型快照
- 内部类型快照
- 动作类型快照
- 动作说明文案快照
- 视频 URL 快照
- 是否支持 AI 监督快照
- 每周频次 `weekly_frequency`
- 单次时长 `duration_minutes`
- 每次组数 `sets`
- 每组次数或总次数 `repetitions`
- 难度 `difficulty`
- 注意事项 `notes`
- 排序 `sort_order`

处方表单根据动作快照的 `action_type_snapshot` 切换展示：

- 时长型动作：突出 `weekly_frequency`、`duration_minutes`、`difficulty`、`notes`；`sets` / `repetitions` 可隐藏或作为可选高级参数。
- 计数型动作：突出 `weekly_frequency`、`sets`、`repetitions`、`difficulty`、`notes`；`duration_minutes` 可作为预计单次时长。

动作库后续改名、调整视频 URL 或停用，不影响已生成处方动作的历史展示和训练记录解释。

## 训练记录规则

`TrainingRecord` 记录一次动作跟练结果，继续属于 `ProjectPatient`，并绑定当时的 `Prescription` 和 `PrescriptionAction`。

患者端模拟提交训练时必须满足：

- 项目患者存在且可访问。
- 项目患者存在当前 active 处方。
- 提交的 `prescription_action` 属于当前 active 处方。
- 处方或动作已被新版本替代时拒绝提交。
- 无 active 处方时不允许提交训练。

训练提交字段初版：

- `prescription_action`
- `training_date`
- `status`：已完成、部分完成、未完成
- `actual_duration_minutes`
- `score`：可选
- `form_data`
- `note`

运动类 `form_data` 建议包含：

- `completed_sets`：完成组数
- `completed_repetitions`：完成次数
- `perceived_difficulty`：主观难度
- `discomfort`：不适情况

其中时长型动作主要依赖通用字段 `actual_duration_minutes`；计数型动作主要依赖 `form_data.completed_sets` 和 `form_data.completed_repetitions`。

## 后端 API

### 固定动作库

`GET /api/prescriptions/actions/?training_type=运动训练&internal_type=motion`

返回只读动作列表，包含动作说明文案、建议参数、视频 URL、是否支持 AI 监督和由 `action_type` 推导出的参数样式。医生角色不提供新增、编辑、删除动作入口。若未来管理员维护动作库，需要另行设计权限和审计策略。

### 医生端处方

`GET /api/prescriptions/?project_patient=<id>`

返回某项目患者处方历史，按版本倒序，内嵌处方动作快照。

`GET /api/prescriptions/current/?project_patient=<id>`

返回某项目患者当前 active 处方；无 active 时返回空状态。

`POST /api/studies/project-patients/<id>/prescriptions/activate-now/`

创建下一版本并立即生效。请求体包含前端看到的当前 active 版本号，以及动作列表和逐动作参数。后端负责生成版本号和快照。

`POST /api/prescriptions/<id>/terminate/`

终止当前 active 处方。前端必须二次确认；后端只允许终止 active 处方。

### 患者端模拟

`GET /api/patient-sim/project-patients/<id>/current-prescription/`

读取当前生效处方及动作列表。第一版仍依赖医生端登录打开，但响应结构按未来患者端设计。

`POST /api/patient-sim/project-patients/<id>/training-records/`

提交单个动作跟练结果，成功后生成一条 `TrainingRecord`。

## 前端信息架构

### 项目患者处方页或处方 Tab

处方入口以 `ProjectPatient` 为上下文，建议放在项目患者详情或研究录入相关页面附近。页面顶部展示患者、项目、分组和当前处方状态。

页面主体：

- 当前处方：展示 active 版本、生效时间、动作表和视频状态。
- 操作区：无 active 时显示“开具处方”；有 active 时显示“调整处方”“终止处方”“打开跟练模拟”。
- 历史版本：展示 archived / terminated 处方，只读查看。

开具或调整处方使用同一个表单：

- 从固定动作库选择动作。
- 每个动作独立编辑处方参数；不同 `action_type` 展示不同表单样式。
- 时长型动作以每周频次、单次时长为主。
- 计数型动作以每周频次、组数、次数为主。
- 新开处方默认使用动作库建议值。
- 调整处方默认复制当前 active 处方动作和参数。
- 提交按钮文案为“保存并立即生效”。

### 固定动作库 Tab

医生只读查看固定运动训练动作。支持按动作类型筛选。每个动作展示：

- 动作名称与分类
- 动作说明文案
- 建议频次/时长
- 是否支持 AI 监督
- 视频 URL 配置状态

不提供新增、编辑、删除按钮。

### 患者端模拟跟练页

模拟页按未来患者端体验组织，但本期由医生端打开。

页面结构：

- 当前处方摘要：版本、生效时间、动作数量。
- 当前处方动作列表：只展示 active 处方中的动作。
- 单动作跟练页：展示视频播放器、动作说明文案、处方参数。
- 训练提交表单：填写完成状态、实际时长、完成组数、完成次数、主观难度、不适情况和备注。

`video_url` 为空时，展示“视频待配置”占位，但允许医生测试处方和训练提交链路。

## 错误处理

- 项目患者不存在或无权限：返回 404/403，前端显示不可访问。
- 无当前生效处方：医生端显示空状态；患者端模拟页显示“暂无可执行处方”。
- 动作库为空：开方表单不可提交，并提示需要系统预置动作。
- 提交处方动作重复：后端拒绝，前端定位到重复动作行。
- 参数非法：后端拒绝，前端在对应字段展示错误。
- 动作已停用：不可用于新处方；历史处方仍可展示。
- 当前 active 版本变化：拒绝本次调整，提示刷新后重试。
- 终止处方后：患者端模拟页不允许继续提交训练。
- 训练提交时动作不属于当前 active 处方：拒绝提交，提示处方已更新。

## 权限

- 医生可以查看固定动作库。
- 医生可以给有权限的项目患者开具、调整、终止处方。
- 医生可以从项目患者处方页打开患者端模拟跟练页。
- 医生不能新增、编辑或删除动作库动作。
- 前端按钮可见性只做 UX；后端接口必须做最终权限裁决。

## 测试策略

### 后端

- 固定动作库可预置 5 个运动训练动作。
- 医生角色不能通过动作库接口新增、编辑、删除动作。
- 创建 v1 处方后状态为 active。
- 调整处方创建 v2 active，并将 v1 归档。
- 处方动作保存动作库关键字段快照。
- 动作库修改后，旧处方动作快照不变。
- 停用动作不能用于新处方。
- 并发调整时 current active 版本不一致会被拒绝。
- 无 active 处方时，患者端模拟训练提交失败。
- 提交非当前 active 处方动作时失败。
- 提交当前 active 处方动作时生成一条 `TrainingRecord`。

### 前端

- 固定动作库 Tab 展示 5 个运动训练动作且没有新增/编辑/删除入口。
- 无处方项目患者可开具处方并立即看到 active v1。
- 有 active 处方时调整处方会复制当前动作参数。
- 调整提交后当前处方变为新版本，历史版本只读可见。
- 终止处方需要二次确认。
- 跟练模拟页只展示当前 active 处方动作。
- `video_url` 为空时显示视频待配置占位。
- 单动作提交训练后显示提交成功，并生成训练记录。
- 无 active 处方时模拟页不展示动作库其它动作。

## 验收点

1. 医生能在固定动作库 Tab 看到 Excel 中 5 个运动训练动作，且不能新增、编辑、删除。
2. 医生给项目患者选择 1 个或多个固定动作并填写参数后，点击“保存并立即生效”生成 v1 active 处方。
3. 医生再次调整处方并提交后，生成 v2 active，v1 进入 archived。
4. 处方动作展示的是快照内容，动作库后续修改不影响旧处方。
5. 患者端模拟页只展示当前 active 处方动作，不展示未被分配的动作库动作。
6. 患者端模拟页可以进入单个动作，查看视频或视频占位、动作说明文案和处方参数。
7. 单个动作跟练提交后生成一条训练记录，并绑定当前 active 处方和处方动作。
8. 无 active 处方、旧版本动作、停用动作均不能提交训练。

## 兼容与风险

- 现有 `Prescription` 状态仍保留 `draft` / `pending` 以兼容旧代码，但本次医生端新流程不暴露草稿和待生效。
- 现有 `frequency` 文本字段可能需要迁移或兼容为结构化 `weekly_frequency`；实施计划应明确迁移策略。
- 现有 `TrainingRecord` 已有 `form_data`，运动类扩展优先放入 JSON，避免为本期模拟训练过度扩表。
- 视频 URL 只是资源引用，不处理上传、鉴权和长期存储策略。
- 患者端模拟页虽然按患者端语义组织，但第一版仍在医生登录上下文中使用，不能等同正式患者端安全模型。
