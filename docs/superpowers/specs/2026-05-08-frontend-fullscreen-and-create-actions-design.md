# 前端：页面满屏 + 新增动作补齐（患者档案、研究项目）设计

日期：2026-05-08  
范围：MotionCare 前端（React + TS + Ant Design）  
目标：修复页面“边缘没有充满屏幕”的留白问题；补齐“患者档案 / 研究项目”相关新增动作，使关键路径可测试、可验收。

## 背景与问题

### 问题 1：页面边缘没有充满屏幕

当前前端几乎没有全局 CSS。浏览器默认会对 `body` 应用 `margin: 8px;`，会导致页面四周出现留白，形成“没有贴满屏幕”的观感。

### 问题 2：新增动作未实现，无法完成测试

用户反馈“所有新增动作都没有实现”，聚焦在：

- 患者档案：列表里的“详情”不可用 / 详情页未接入 API / 路由缺失
- 研究项目：项目详情页的“项目患者”缺少“添加患者到项目”入口与交互闭环

## 目标与非目标

### 目标（最小闭环）

- **布局满屏**：消除默认留白，保证根节点高度铺满
- **患者档案**：
  - 列表页“详情”可跳转到 `/patients/:patientId`
  - 详情页接入 `GET /api/patients/:id/` 展示基础信息，并有 loading / error 状态
- **研究项目**：
  - 在项目详情的“项目患者”Tab 中提供“添加患者到项目”
  - 通过弹窗选择全局患者并提交，成功后刷新项目患者表格

### 非目标（本次不做）

- 患者详情编辑/删除、跨模块联动（访视/训练/处方等）
- 项目患者移除、批量加入、复杂筛选
- 更大范围的 UI 重构或样式体系建设

## 方案选择（已确认）

### 布局修复方案：A（已选）

新增全局样式文件并在入口引入，设置：

- `body { margin: 0; }`（清掉默认留白）
- `html, body, #root { height: 100%; }`（根高度铺满，避免内部 `minHeight: 100vh` 与外层不一致）

该方案对现有页面侵入最小且覆盖面完整（包括登录页等非 `AdminLayout` 页面）。

### 新增动作补齐范围：最小可用闭环（推荐并采用）

优先打通“能创建/能进入详情/能加入项目”三条关键路径，保证可测可验收；不引入大范围 CRUD。

## 设计细节

### 1) 全局满屏样式

#### 变更点

- 新增：`frontend/src/styles/global.css`（或同等路径）
- 修改：`frontend/src/main.tsx` 引入该样式

#### 样式内容（概念）

- `html, body, #root { height: 100%; }`
- `body { margin: 0; }`

（如需要统一背景，可额外设置 `body { background: ... }`，但本次不强制。）

### 2) 患者档案：详情可用 + 详情页接 API

#### 路由

在 `frontend/src/app/App.tsx` 增加路由：

- `GET /patients/:patientId` → `PatientDetailPage`

#### 患者列表页交互

在 `PatientListPage` 的表格“操作”列：

- “详情”从无动作改为跳转
  - 方案：使用 `Link` 或 `navigate`
  - 路由：`/patients/${row.id}`

#### 详情页数据与展示

`PatientDetailPage`：

- 参数：
  - `patientId` 来自 `useParams()`
  - 解析为数字；非法则展示错误提示（例如 `Alert`/`Card` 文案）
- 请求：
  - `GET /api/patients/${id}/`
  - 注意前端 axios baseURL 为 `"/api"`，代码中保持 `"/patients/${id}/"` 形式即可
- 状态：
  - loading：显示 `Card loading` 或 `Spin`
  - error：提示“患者不存在或无权限访问”
- 展示字段（最小集）：
  - 姓名、性别、年龄、手机号、主治医生（如后端返回为 ID 或对象，按实际 serializer 字段展示）

### 3) 研究项目：项目患者 Tab 增加“添加患者到项目”

#### 入口与交互形态

在 `ProjectPatientsTab`：

- `Card` 增加 `extra` 按钮：“添加患者”
- 点击打开 `Modal`

#### 候选患者来源

- 通过 `GET /api/patients/` 获取全局患者列表
- 使用 `Select`（可搜索）展示为 `姓名（手机号）`
- 初期仅支持单选（后续可扩展多选）

#### 提交接口

提交调用：

- `POST /api/studies/project-patients/`
- body：
  - `{ project: projectId, patient: patientId }`

成功后：

- 关闭弹窗、清空选择
- `invalidateQueries(["project-patients", projectId])` 刷新表格

失败提示：

- toast 显示友好错误（优先使用后端 `detail` 字段，否则兜底文案）

## 错误处理与权限假设

### 权限/登录

前端通过 `RequireAuth` + `AuthProvider` 获取 `/api/me/`。

- 若会话校验失败或未登录：跳转 `/login`
- 本次变更不修改权限策略，仅保证新增动作与详情页在已有权限下可用

### 典型失败场景

- `GET /patients/:id` 返回 404/403：展示“项目不存在或无权限访问”同类错误提示
- `POST /project-patients/`：
  - 重复加入：后端可能 400/409（以实际为准），前端展示错误 toast

## 测试与验收点（可操作）

### 布局

- 访问任意页面（含登录页/后台页），页面四周不再有默认留白

### 患者档案

- 在“患者档案”点击“新建患者”创建成功后，列表刷新可见
- 点击某行“详情”能进入详情页
- 详情页能显示基础字段；并有 loading/error 状态

### 研究项目

- 在“研究项目”创建项目成功后，列表刷新可见
- 进入项目详情 → “项目患者”Tab
- 点击“添加患者”，选择一名全局患者并提交后，表格出现新增行

## 影响面与回滚

- 影响面主要在前端路由、两个页面与一个全局样式文件
- 回滚可通过撤销新增 CSS 引入与相关页面/路由改动完成；不涉及 DB 迁移
