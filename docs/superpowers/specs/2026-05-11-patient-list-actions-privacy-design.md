# 患者列表操作与隐私展示设计

日期：2026-05-11  
范围：MotionCare 管理端患者列表页、患者 API 序列化字段。

## 背景

当前患者列表操作列同时提供「编辑」与「详情」，详情入口只存在于操作列；列表展示 `primary_doctor` ID，不利于业务识别；手机号以明文展示，不符合列表场景的最小暴露原则。详情页已经具备患者删除入口、二次确认和后端删除保护，本次需求是在列表页补齐删除入口，并让列表行本身更便于进入详情。

## 选定方案

采用「前端列表轻量增强 + 后端补医生姓名字段」方案。

- 患者姓名在列表中渲染为详情链接，点击进入 `/patients/:patientId`。
- 操作列移除「详情」，保留「编辑」，新增「删除」。
- 列表手机号脱敏显示为 `前三位 + **** + 后四位`；详情页和编辑弹窗仍显示完整手机号，便于核对和维护。
- 列表移除「主治医生 ID」列，改为「主治医生」列，展示后端返回的 `primary_doctor_name`。若无主治医生则显示 `—`。
- 后端 `PatientSerializer` 增加只读字段 `primary_doctor_name`，保留既有 `primary_doctor` ID 字段用于创建、编辑和兼容现有调用。

## 交互设计

患者列表表格列调整为：姓名、性别、年龄、手机号、主治医生、操作。姓名列是主详情入口，操作列不再提供「详情」文本链接，减少重复入口。

列表删除按钮采用危险操作样式。点击后打开阻断式确认弹窗：

- 若患者没有研究项目关联，展示永久删除且不可恢复的影响说明，用户确认后调用 `DELETE /api/patients/:id/`。
- 若患者仍有关联研究项目，弹窗展示禁止删除原因，并提示需先到项目中删除或解绑该患者；此时不发起删除请求。
- 若前端预检未发现关联但后端仍返回删除失败，弹窗展示后端 `detail`，保持与详情页删除保护一致。

删除成功后刷新患者列表。删除失败不关闭弹窗，便于用户阅读阻断原因。

## 数据流

后端患者列表接口继续使用 `/api/patients/`。序列化输出增加：

- `primary_doctor_name`: 字符串或 `null`，来源于 `Patient.primary_doctor.name`。

前端 `PatientRow` 类型增加 `primary_doctor_name`，手机号脱敏只在渲染层处理，不改变 API 原始值。删除前端可复用详情页已有错误解析逻辑和删除确认组件，以保持交互文案风格一致。

## 错误处理

删除按钮必须先进入确认弹窗，不允许单击直接删除。有关联项目的患者在弹窗中显示阻断原因，文案明确包含「需先到项目中删除/解绑患者」。后端返回的其他错误通过同一弹窗展示；创建、编辑等原有错误处理不在本次范围内调整。

手机号脱敏函数需要兼容异常输入：长度不足 7 位时，保留首尾各 1 位，中间用 `*`；空值显示 `—`。该兼容仅影响列表展示，不修改存储值。

## 测试与验收

- 后端测试：患者序列化结果包含 `primary_doctor_name`，且值为主治医生姓名。
- 前端测试或手工验收：患者姓名可点击进入详情；操作列无「详情」入口；操作列有「删除」按钮。
- 前端测试或手工验收：列表手机号按规则脱敏，编辑弹窗和详情页仍能显示完整手机号。
- 前端测试或手工验收：主治医生列显示姓名而不是 ID。
- 前端测试或手工验收：无项目关联患者可二次确认删除，成功后列表刷新；有关联项目患者在弹窗中提示需先到项目中删除/解绑患者。

## 非目标

- 不改变患者详情页删除、停用、编辑的业务规则。
- 不新增患者搜索、筛选或分页能力。
- 不隐藏 API 中的手机号原始值；本次只调整管理端列表展示。
- 不删除或改名 `primary_doctor` ID 字段，以免影响创建和编辑流程。
